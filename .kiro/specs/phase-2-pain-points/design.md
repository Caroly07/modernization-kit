# Phase 2: Pain Point Assessment — Design

## Approach

This phase requires direct conversation with actual users, not just the technical team. The domain expert facilitates introductions or relays information from users.

## Interview Technique

- Ask open-ended questions, not yes/no
- "Walk me through your typical day" reveals more than "what's broken"
- Follow up on workarounds — they reveal the gap between what the system does and what users need
- Ask about the "magic feature" — it reveals unspoken assumptions about what's possible
- Listen for emotional language ("I hate when...", "It drives me crazy that...") — these are the high-impact pain points

## Pain Point Categories

| Category | Examples |
|---|---|
| Performance | Slow page loads, timeouts, batch processing delays |
| Usability | Too many clicks, confusing navigation, lost scroll position |
| Missing features | No mobile access, no real-time updates, no self-service |
| Data quality | Duplicates, stale data, inconsistent formats |
| Integration gaps | Manual re-entry between systems, no API access |
| Reporting | Can't get the data they need without Excel exports |
| Workflow | Approval bottlenecks, no notifications, manual handoffs |
| Security/access | Too much access, not enough access, shared accounts |

## Output Priority Matrix

Rank pain points using: Impact × Frequency × Users Affected = Priority Score

High-priority pain points drive Phase 3 architecture decisions and Phase 4 phase ordering.
